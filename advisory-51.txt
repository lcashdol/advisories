The PrimeBase Database Server supports all common database access standards (PBT, SQL, ODBC, JDBC, PHP, Perl, RealBasic, EOF and DAL) and protocols (TCP/IP, Shared Memory and Appletalk)&lt;/span&gt;&lt;/td&gt;&lt;/tr&gt;&lt;tr&gt;&lt;td style="background-color: rgb(220,220,220);"&gt;&lt;b&gt;Vulnerability:&lt;/b&gt;&lt;div&gt;
Local attackers can exploit these vulnerabilities to clobber root owned system files and modify software binaries. This could possibly lead to a denial of service or system compromise.
</p><ol>
  <li>Poor use of temporary files during installation.
I noticed the PrimeBase install script creates the following files in /tmp:</li></ol>
<p>[nobody $] ln -s /etc/shadow /tmp/PrimeBase.log
</p><p>Then if a malicious user has previous knowledge of the administrators installation of PrimeBase the contents of /etc/shadow will be overwritten with the contents of PrimeBase.log.
</p><p>LOG=&amp;quot;/tmp/PrimeBase.log&amp;quot;
</p><p>echo &amp;quot;$str:[y/n]&amp;quot; | tee $LOG<br/>
echo &amp;quot;PrimeBase Installation: $now&amp;quot; &amp;gt;&amp;gt; $LOG
</p><p>2. Poor default file permissions.<br/>
A malicious local user could manipulate the binaries for PrimeBase used by the administrator and execute arbitrary code. The attacker would need to wait until the Database was restarted or the system rebooted.
</p><p>root@Fester local]# ls -ld /usr/local/primebase
